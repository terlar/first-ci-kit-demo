# first-ci-kit demo

A realistic platform-engineering IaC demo for [first-ci-kit] — a Nix library
that generates GitHub Actions workflows and GitLab CI pipelines from a single
declarative Nix configuration.

The repository models six infrastructure stacks managed with [OpenTofu],
deployed to two environments (`dev`, `stg`). The same `dev/ci.nix` source
produces both CI platforms with correct cross-stack job ordering on each.

## Repository structure

```
dev/
  ci.nix              # Pipeline definitions (source of truth for both platforms)
  stacks.nix          # Stack/component/deployment topology and cross-stack needs
  flake.nix           # Dev partition flake; declares first-ci-kit input
  flake-module.nix    # Pre-commit hooks that regenerate CI files from ci.nix
terraform/
  <stack>/<component>/  # OpenTofu modules, mirroring the stack topology
  modules/component/    # Shared component module
  deployments/          # Per-environment .tfvars and .tfbackend files
.github/workflows/
  ci.yml              # Generated — top-level orchestration workflow
  profile-tofu.yml    # Generated — reusable plan+deploy workflow
.gitlab-ci.yml        # Generated — top-level pipeline with include: entries
gitlab-templates/
  profile-tofu/
    template.yml      # Generated — GitLab CI component template for plan+deploy
```

> **Generated files** (`.github/workflows/`, `.gitlab-ci.yml`,
> `gitlab-templates/`) are never edited by hand. They are regenerated by
> running the `first-ci-kit-gen-github-actions` and
> `first-ci-kit-gen-gitlab-ci` pre-commit hooks:
>
> ```
> nix develop dev -c pre-commit run --all-files
> ```

## Stacks and cross-stack dependencies

Six stacks, each with two or three components, all deployed to `dev` and `stg`:

```
networking  ──────────────────────────────────────────┐
  vpc                                                  │
  dns                                                  │
                                                       ▼
security                          cluster ─────────────────────────────┐
  iam ──────────────────────┐       control-plane (needs: networking)   │
  certs ────────────────┐   │       node-pools    (needs: control-plane)│
                        │   │                                           │
                        │   └──► database                               │
                        │          postgres (needs: networking, iam)    │
                        │          redis    (needs: networking)         │
                        │                                               │
                        └──────► platform-services ◄───────────────────┘
                                   ingress       (needs: cluster, certs)
                                   cert-manager  (needs: cluster, dns)

                                 observability  ◄─────────────────────┘
                                   metrics   (needs: cluster)
                                   logging   (needs: cluster)
                                   alerting  (needs: metrics)
```

Cross-stack `needs` are declared once in `dev/stacks.nix` and automatically
translated into `needs:` entries in GitHub Actions and `plan_needs` inputs in
GitLab CI — no manual wiring in the generated files.

Staging (`stg`) jobs additionally wait for all `dev` jobs to complete first.

## The `profile-tofu` pipeline

Each `(stack, component, deployment)` triple runs through the same two-job
pipeline:

| Job | Steps |
|---|---|
| **plan** | `tofu init` → `tofu validate` → `tofu plan` → upload `tfplan` artifact |
| **deploy** | download artifact → `tofu init` → `tofu apply -auto-approve tfplan` |

Both jobs activate the per-stack Nix dev shell (`nix print-dev-env
.#profile-tofu`) which provides OpenTofu and any other required tooling.
Backend config and var files are loaded from `terraform/deployments/`.

On GitHub Actions the deploy job is skipped on pull requests (change detection
controls which jobs run at all). On GitLab CI the plan job waits for its
upstream `_deploy` jobs via the `plan_needs` component input before starting.

## How it works

```
dev/ci.nix  ──► first-ci-kit  ──► .github/workflows/ci.yml
                                   .github/workflows/profile-tofu.yml
                                   .gitlab-ci.yml
                                   gitlab-templates/profile-tofu/template.yml
```

`dev/ci.nix` declares two pipelines using the `first-ci-kit` Nix module:

- **`profile-tofu`** — the reusable plan+deploy pipeline template, with
  per-platform rendering (reusable workflow on GitHub Actions, GitLab CI
  component on GitLab CI)
- **`default`** — iterates over `dev/stacks.nix` to generate one
  `pipelineCall` job per `(stack, component, deployment)` triple, wiring
  up change detection, `needs` ordering, and rules
